# Lucid market and documentation benchmark

**Prepared:** 21 July 2026

**Audit target:** workspace snapshot `10369f3`, based on `origin/master` at `c21990b`

**External-source rule:** official documentation, official source repositories, and formal specifications only

**Purpose:** test whether Lucid is explained accurately, whether every important product journey has a complete page, and where its documentation falls behind the strongest adjacent products and protocols.

## Executive conclusion

Lucid has a credible and differentiated category: **a TypeScript application runtime for machine commerce**. Its strongest product story is not “another agent framework” or “another x402 SDK.” It is the application transaction around a paid capability: typed input and output, payment or credential admission, policy, idempotency, fulfillment, task ownership, settlement, and durable recording behind one adapter-independent runtime. The current [documentation home](../../lucid-docs/content/docs/index.mdx), [architecture](../ARCHITECTURE.md), [x402 support matrix](../../lucid-docs/content/docs/protocols/x402.mdx), [seller quickstart](../../lucid-docs/content/docs/start/sell-paid-api.mdx), and [budgeted-buyer quickstart](../../lucid-docs/content/docs/start/budgeted-buyer.mdx) already express most of that position.

The documentation now has a broad outcome-led skeleton, but broad navigation is not the same as complete coverage. The local tree contains 78 MDX pages; a heuristic content audit found 49 with fewer than 150 prose words, 40 with no second-level heading, 56 with no external primary-source link, and 43 with no internal next-step link. The complete inventory is visible under the [documentation content tree](../../lucid-docs/content/docs/). These counts do not prove a page is bad—short boundary pages can be intentional—but they identify a site where many destinations are summaries rather than full guides.

The most urgent issue is accuracy. The [A2A package page](../../lucid-docs/content/docs/packages/a2a.mdx) describes Lucid-owned `/tasks` routes and a custom task model as “A2A,” while official A2A 1.0 defines different operations, routes, version negotiation, and data models in its [latest specification](https://a2a-protocol.org/latest/specification/) and [v1 migration notes](https://a2a-protocol.org/latest/whats-new-v1/). The [wallet page](../../lucid-docs/content/docs/packages/wallet.mdx) and [identity page](../../lucid-docs/content/docs/packages/identity.mdx) also contain Next API examples that do not match the current [wallet contracts](../../packages/types/src/wallets/index.ts) and [identity exports](../../packages/identity/src/index.ts). The [AP2 package page](../../lucid-docs/content/docs/packages/ap2.mdx) pins a v0.1 descriptor even though the official project released [AP2 v0.2 on 28 April 2026](https://github.com/google-agentic-commerce/AP2/releases/tag/v0.2.0). These are stop-ship documentation defects because a reader can reasonably infer interoperability or call APIs that are not present.

The market benchmark points to four priorities:

1. **Correct and qualify protocol claims before adding more pages.** Publish exact conformance profiles for x402, MPP, A2A, ERC-8004, and AP2. “Supports” must mean a named version, binding, feature set, and tested compatibility target.
2. **Make documentation executable beyond two quickstart fixtures.** The current [documentation test script](../../scripts/docs-snippets.test.ts) protects the Stable seller/buyer sources and homepage example, while most package, protocol, deployment, and example snippets remain handwritten.
3. **Turn operational checklists into production runbooks.** The [production checklist](../../lucid-docs/content/docs/operate/production-checklist.mdx) and [security page](../../lucid-docs/content/docs/operate/security.mdx) identify the right concerns, but they do not yet show provider deployment, crash-window recovery, reconciliation, tenant isolation, threat boundaries, rollback, or incident procedures end to end.
4. **Document composition with agent frameworks instead of competing with them.** OpenAI Agents SDK, Vercel AI SDK, Mastra, and LangGraph own model loops, tools, memory, orchestration, and evaluation. AgentKit owns wallet/onchain actions. Lucid should show how those systems call or expose a Lucid-paid service and reserve its category claim for machine-commerce application semantics. The products’ own scope descriptions support this division: [OpenAI Agents SDK](https://developers.openai.com/api/docs/guides/agents), [Vercel AI SDK](https://ai-sdk.dev/docs/introduction), [Mastra](https://mastra.ai/docs), [LangGraph](https://docs.langchain.com/oss/javascript/langgraph/overview), and [Coinbase AgentKit](https://docs.cdp.coinbase.com/agent-kit/welcome).

### Post-audit remediation in this workspace

This report remains a point-in-time audit of snapshot `10369f3`; its counts and
appendix are preserved as the baseline. The documentation work following the
audit has:

- corrected wallet and identity API examples against their public source
  contracts;
- replaced unqualified A2A/AP2/MPP/ERC-8004 claims with versioned compatibility
  profiles and explicit unsupported surfaces;
- separated Stable npm from the repository's Next package set;
- expanded seller, buyer, payment lifecycle, retry, discovery, scheduler,
  security, storage, observability, facilitator, and adapter deployment pages;
- added current framework-composition and MCP-boundary guides;
- classified unavailable hosted destinations as boundary pages; and
- added a content check requiring pages under 150 words to be declared as an
  intentional index/boundary or expanded.

Remaining product work is tracked by the P1/P2 acceptance criteria below,
especially generated API reference, executable coverage beyond the core
quickstarts, official conformance suites, durable task/scheduler adapters, and
full hosted-product contracts if those services become publicly available. The
blank template passes the five-adapter packed-workspace matrix, but the two
legacy trading templates still generate pre-v3 source/dependency wiring; their
docs are marked migration-required until implementation and verification are
added.

Two generator defaults also need a separate product decision: the identity
wizard currently enables on-chain auto-registration by default, and several
templates still seed a legacy development facilitator URL. Generated guides
now tell readers to review those values, but documentation cannot substitute
for safer defaults and provider verification in the generator itself.

## Recommended positioning contract

### Primary message

> **Turn a typed TypeScript capability into a discoverable paid service, with authorization, fulfillment, retries, tasks, and durable state governed as one application transaction.**

This is a recommendation synthesized from the [current homepage](../../lucid-docs/content/docs/index.mdx), [canonical HTTP authorization transaction](../../lucid-docs/content/docs/build/payment-model.mdx), and [runtime architecture](../ARCHITECTURE.md). It is deliberately narrower than “agent framework” and broader than “payment middleware.”

### What Lucid is

- A typed extension runtime and canonical entrypoint registry, documented in [core](../../lucid-docs/content/docs/packages/core.mdx) and [HTTP](../../lucid-docs/content/docs/packages/http.mdx).
- An application layer around x402 or MPP admission, policy, fulfillment, settlement, tasks, and storage, documented in the [payment model](../../lucid-docs/content/docs/build/payment-model.mdx) and [x402 application transaction](../../lucid-docs/content/docs/protocols/x402.mdx).
- A seller-and-buyer toolkit with explicit buyer recipient and spend controls, demonstrated by the [seller](../../lucid-docs/content/docs/start/sell-paid-api.mdx) and [buyer](../../lucid-docs/content/docs/start/budgeted-buyer.mdx) paths.
- A framework-portable service runtime with Hono, Express, Next.js-generated, and TanStack integration paths under [deployment](../../lucid-docs/content/docs/operate/deploy/index.mdx).

### What Lucid is not

- It is not a model-provider abstraction, LLM loop, or general agent orchestration framework. Those jobs are explicitly central to [Vercel AI SDK](https://ai-sdk.dev/docs/agents/overview), [OpenAI Agents SDK](https://developers.openai.com/api/docs/guides/agents), [Mastra](https://mastra.ai/docs), and [LangGraph](https://docs.langchain.com/oss/javascript/langgraph/overview).
- It is not a wallet, facilitator, settlement network, or payment protocol. The [Lucid home page](../../lucid-docs/content/docs/index.mdx) already says these remain external layers.
- An Agent Card is not x402 Bazaar publication; the [Lucid A2A concept page](../../lucid-docs/content/docs/protocols/a2a.mdx) and [x402 Bazaar documentation](https://docs.x402.org/extensions/bazaar) describe different discovery surfaces.
- ERC-8004 registration is not proof that an agent is safe, correct, or available. The formal [EIP-8004 security considerations](https://eips.ethereum.org/EIPS/eip-8004#security-considerations) warn about Sybil behavior and the limits of advertised metadata.
- Lucid’s current AP2 package is role metadata, not checkout, mandate, authorization, or settlement execution, as the [local AP2 page](../../lucid-docs/content/docs/packages/ap2.mdx) correctly states. The version it advertises is nevertheless stale relative to [AP2 v0.2](https://ap2-protocol.org/ap2/specification/).

## Methodology

### Local audit

The audit covered every page family under [Lucid docs](../../lucid-docs/content/docs/), the [root README](../../README.md), package READMEs and public types where claims needed verification, and documentation quality controls declared in the [root scripts](../../package.json). It compared prose with the current source contracts for wallet, identity, A2A, AP2, HTTP routes, and package manifests rather than treating built MDX as proof of API correctness.

Page-depth measurements were heuristic. Frontmatter, code, tables, and prose have different value, so word and heading counts are triage signals rather than quality scores. A page was judged “full” only when it covered the relevant parts of the full-page contract below.

### External benchmark

Only first-party or formal sources were used. The comparison set covers:

No community article, comparison blog, aggregator, or search-result summary was used as evidence. “Not evidenced” means no first-party page was located during this review; it does not prove a capability is absent from code or a private product.

### Rating legend

| Rating            | Meaning                                                                                                               |
| ----------------- | --------------------------------------------------------------------------------------------------------------------- |
| **Strong**        | First-party, discoverable, end-to-end guidance with runnable examples or complete reference.                          |
| **Partial**       | First-party coverage exists but leaves material setup, API, failure, security, or production questions unresolved.    |
| **External**      | The capability is shown through an ecosystem integration or adjacent product, not the core surface being benchmarked. |
| **Boundary**      | The product intentionally does not own this job and explains that boundary.                                           |
| **Not evidenced** | No current first-party page was located; no implementation conclusion should be inferred.                             |

## Detailed market findings

### 1. Upstream x402 and Nevermined: separate protocol, runtime, and platform

The official [x402 introduction](https://docs.x402.org/introduction) presents a protocol ecosystem of buyers, sellers, wallets, facilitators, schemes, networks, and extensions. Its [buyer](https://docs.x402.org/getting-started/quickstart-for-buyers) and [seller](https://docs.x402.org/getting-started/quickstart-for-sellers) quickstarts are intentionally payment-focused. The [facilitator page](https://docs.x402.org/core-concepts/facilitator) explains verify/settle ownership and distinguishes testnet public infrastructure from production choice. This is the layer Lucid implements and surrounds; it is not a competitor category that Lucid should blur into its own runtime.

Nevermined is the more direct commercial comparison. Its [welcome page](https://docs.nevermined.app/docs/getting-started/welcome) starts from monetization use cases and exposes agent-to-agent payments, MCP monetization, static assets, cost/margin observability, fiat, stablecoins, SDKs, REST, and a dashboard. The [quickstart](https://docs.nevermined.app/docs/getting-started/quickstart) takes both builder and subscriber through hosted account/API-key, plan registration, payment middleware, purchase, and credential use. The [registration guide](https://docs.nevermined.app/docs/development-guide/registration) then expands plan types, endpoint registration, and security patterns rather than forcing that detail into the first page.

Nevermined also publishes an explicit specification layer. Its [technical-specification index](https://docs.nevermined.app/docs/specs/introduction) says normative specs are the source of truth and lists x402 smart-account and card-delegation extensions. The [Nevermined x402 guide](https://docs.nevermined.app/docs/development-guide/nevermined-x402) explains how its ERC-4337/session-key/delegation settlement differs from a basic x402 transfer and walks through subscriber, agent, facilitator, verification, execution, and settlement roles.

**Lesson for Lucid:** add a “Runtime or monetization platform?” decision page. State that Lucid gives developers application-owned runtime behavior and provider portability; list managed plans, fiat checkout, credit issuance, payout management, and operator dashboard as hosted-product or external-platform responsibilities unless Lucid actually ships them. Match Nevermined’s separation of Guides, API/SDK reference, payment patterns, components, and formal Specs without copying its full-platform claim.

### 2. OpenAI Agents SDK: teach control boundaries early

OpenAI’s strongest documentation lesson is not volume; it is the decision map. The [Agents SDK overview](https://developers.openai.com/api/docs/guides/agents) tells readers where to begin, distinguishes the SDK from lower-level API use, and branches into models, running, orchestration, guardrails, results, state, tools, MCP, and observability. The [quickstart](https://developers.openai.com/api/docs/guides/agents/quickstart) adds complexity incrementally rather than presenting a complete architecture before first success.

The safety material names where controls execute. [Guardrails and approvals](https://developers.openai.com/api/docs/guides/agents/guardrails-approvals) distinguishes input, output, tool, and approval boundaries and explains resumable approval state. [Integrations and observability](https://developers.openai.com/api/docs/guides/agents/integrations-observability) distinguishes hosted/public MCP from runtime-managed private MCP and describes tracing scope.

**Lesson for Lucid:** every payment or task control should say whether it runs before verification, before reservation, before fulfillment, before settlement, or after recording. The [payment model](../../lucid-docs/content/docs/build/payment-model.mdx) introduces the transaction, but the site needs one normative state diagram and a failure table shared by invoke, stream, and tasks.

### 3. Vercel AI SDK: framework-specific first success plus deterministic tests

Vercel’s docs separate universal foundations from framework-specific starts, agents, Core, UI, reference, migrations, and troubleshooting on the [introduction](https://ai-sdk.dev/docs/introduction). The [Next.js quickstart](https://ai-sdk.dev/docs/getting-started/nextjs-app-router) completes an application route and streamed UI rather than stopping at a library call. Dedicated [testing](https://ai-sdk.dev/docs/ai-sdk-core/testing), [telemetry](https://ai-sdk.dev/docs/ai-sdk-core/telemetry), and [DevTools](https://ai-sdk.dev/docs/ai-sdk-core/devtools) pages explain deterministic mocks, opt-in tracing, input/output capture, and local-only sensitive-data risk.

Vercel also demonstrates a compelling adjacent commerce story through its official [x402 AI starter](https://vercel.com/templates/next.js/x402-ai-starter) and [402-mcp announcement](https://vercel.com/changelog/402-mcp-enables-x402-payments-in-mcp). That material shows that the market increasingly expects paid tools to compose with MCP, even when payment is not part of the core agent SDK.

There is a useful caution about release truth: when retrieved on 21 July 2026, the official docs navigation labeled “v6 (Latest)” on the [introduction](https://ai-sdk.dev/docs/introduction), while Vercel’s official [AI SDK 7 announcement](https://vercel.com/blog/ai-sdk-7) was dated 25 June 2026. Lucid’s visible [Stable/Next contract](../../lucid-docs/content/docs/reference/release-channels.mdx) is therefore a genuine strength, but it must be backed by automated claim verification to avoid the same documentation/release drift.

**Lesson for Lucid:** compile every public TypeScript example, provide test doubles for facilitator and wallet behavior, and add framework-specific paid-service recipes that end in an actual expected response—not just adapter binding snippets.

### 4. Mastra: production is a top-level documentation job

Mastra gives production equal navigation weight with development. The official [docs home](https://mastra.ai/docs) exposes agents, workflows, long-running capabilities, server, observability, evals, and deployment. The [deployment overview](https://mastra.ai/docs/deployment/overview) distinguishes standalone server, existing-framework, managed platform, deployer, and self-managed paths. [Observability](https://mastra.ai/docs/observability/overview) connects traces, logs, metrics, feedback, storage, and sensitive-data filtering.

The security pages also separate authentication from authorization. [Server authentication](https://mastra.ai/docs/server/auth) explains which routes become protected, while [middleware](https://mastra.ai/docs/server/middleware) warns that authentication alone does not prevent cross-user access and shows resource scoping. Mastra’s official [A2A announcement](https://mastra.ai/blog/introducing-agent-to-agent-support) and [MCP guide](https://mastra.ai/docs/agents/mcp-guide) make interoperability a discoverable product surface rather than an incidental example.

**Lesson for Lucid:** expand “Operate” into concrete deployment and multi-tenant runbooks. Payment authorization is not application authorization; wallet identity, task ownership, service tenancy, and operator roles require a clear resource-scoping model.

### 5. LangGraph/LangSmith: explain the product stack and shared responsibility

LangChain’s documentation explicitly differentiates framework, runtime, harness, and platform roles: LangChain is the agent framework, LangGraph the orchestration runtime, and LangSmith the tracing/evaluation/deployment platform on the [LangGraph overview](https://docs.langchain.com/oss/javascript/langgraph/overview). That taxonomy prevents one product name from silently absorbing every adjacent responsibility.

Production coverage is unusually complete. The [Agent Server API reference](https://docs.langchain.com/langsmith/server-api-ref) enumerates assistants, threads, runs, crons, store, A2A, MCP, and system endpoints. [Self-hosting](https://docs.langchain.com/langsmith/self-hosted) documents infrastructure dependencies and deployment shapes. [Data storage and privacy](https://docs.langchain.com/langsmith/data-storage-and-privacy) explains where state lives and how tracing/telemetry can be disabled. The [shared-responsibility model](https://docs.langchain.com/langsmith/shared-responsibility-model) makes provider and customer obligations explicit.

**Lesson for Lucid:** publish one stack diagram that separates OSS runtime, hosted Lucid products, facilitators, wallet providers, networks, protocol registries, and the operator’s application. Then give each boundary an owner, stored data, credentials, failure mode, and support path.

### 6. Coinbase AgentKit/CDP: the nearest adjacent commerce benchmark

AgentKit positions itself as a wallet and onchain action toolkit that composes with frameworks, not as the whole agent runtime. The [architecture](https://docs.cdp.coinbase.com/agent-kit/core-concepts/architecture-explained) separates Action Providers, Wallet Providers, and framework extensions; the [quickstart](https://docs.cdp.coinbase.com/agent-kit/getting-started/quickstart) generates choices for LangChain, Vercel AI SDK, OpenAI Agents SDK, or MCP depending on language.

CDP’s x402 documentation is the closest documentation benchmark for Lucid’s commercial journeys. The [seller quickstart](https://docs.cdp.coinbase.com/x402/quickstart-for-sellers) covers facilitators, framework examples, schemes, test flow, errors, networks, and production setup. Separate pages cover the [buyer](https://docs.cdp.coinbase.com/x402/quickstart-for-buyers), [Bazaar discovery](https://docs.cdp.coinbase.com/x402/bazaar), and [MCP server payment bridge](https://docs.cdp.coinbase.com/x402/mcp-server). Wallet safety is documented through a fail-secure [policy engine](https://docs.cdp.coinbase.com/wallets/security-and-policies/policy-engine/overview) and [security overview](https://docs.cdp.coinbase.com/wallets/security-and-policies/security-overview).

**Lesson for Lucid:** lead with the application-transaction advantage that CDP does not claim, but match CDP’s clarity on facilitator choice, supported combinations, errors, production credentials, wallet policy, and the boundary between a payment protocol and a provider implementation.

| Payment negotiation and execution   | x402 v2 defines its challenge/payment transport and schemes in the [v2 spec](https://github.com/x402-foundation/x402/blob/main/specs/x402-specification-v2.md); MPP-related Payment Authentication remains an [active Internet-Draft](https://datatracker.ietf.org/doc/draft-ryan-httpauth-payment/). | Claim only tested schemes, methods, networks, assets, and wire revisions. Lucid-owned budgets, policies, reservations, and records should be named as Lucid behavior. |
| Commerce authorization and evidence | AP2 v0.2 defines mandates, receipts, verification, and dispute evidence in the [AP2 specification](https://ap2-protocol.org/ap2/specification/).                                                                                                                                                      | Claim role metadata today. Do not imply mandate issuance, deterministic verification, payment authorization, checkout, or dispute evidence.                           |

No one protocol supplies the complete commerce runtime. This layered explanation is the clearest justification for Lucid’s extension architecture, provided each layer’s implementation status is explicit.

Two site-level rules follow from this inventory:

1. A **Stable** page should be generated and tested against the npm versions in one declared compatibility set, not against the repository workspaces. The current [Stable quickstart harness](../../scripts/test-docs-stable-quickstart.ts) is the correct pattern to expand.

## The full-page contract

A critical documentation destination should not be considered complete until it contains every applicable element below. Short index and boundary pages are exempt, but they must route the reader to a complete destination.

1. **Outcome and non-goals** — one sentence describing what the reader will achieve and what the feature does not provide.
2. **Status and compatibility** — Stable/Next/Experimental, package version, protocol version/draft, adapter, runtime, network, asset, and provider combinations.
3. **Prerequisites** — install command, credentials, funded testnet requirement, environment, storage, and external-service account.
4. **Runnable path** — source-linked code that is compiled or executed in CI.
5. **Expected evidence** — command output, HTTP status/headers/body, task transition, settlement/receipt identifier, or UI state.
6. **Complete API/configuration surface** — options, defaults, return types, errors, and links to generated symbols.
7. **Failure and retry semantics** — malformed input, policy rejection, verification failure, timeout, duplicate, handler failure, settlement failure, recording failure, and process loss.
8. **Security and privacy** — trust boundary, secrets, replay, tenant/resource authorization, logging/redaction, abuse limits, and protocol-specific threats.
9. **Production and durability** — required durable ports, consistency/atomicity, scale topology, provider limits, health, observability, reconciliation, rollback, and incident path.
10. **Testing and conformance** — local fixture, fake provider, crash injection, protocol vector/TCK, adapter matrix, and clean-room installation.
11. **Next steps** — one primary next page, related reference, upstream formal source, and migration/changelog route.

This contract should be encoded in docs linting. A package or protocol page can declare an element “not applicable,” but it should not silently omit it.

- **Vercel AI SDK:** the official docs displayed “v6 (Latest)” at retrieval on the [introduction](https://ai-sdk.dev/docs/introduction), while the official [AI SDK 7 post](https://vercel.com/blog/ai-sdk-7) is dated 25 June 2026. The benchmark uses the docs for information architecture and the dated post only for v7 release facts.
- **Mastra:** the docs displayed “Latest Version” but did not expose a numeric per-page version or update date on the pages reviewed. Dated product facts are limited to official posts such as the [A2A announcement](https://mastra.ai/blog/introducing-agent-to-agent-support).
- **LangGraph/LangSmith:** the reviewed guide pages did not consistently expose per-page dates. Version-specific server claims should be verified against the official [Agent Server changelog](https://docs.langchain.com/langsmith/agent-server-changelog) when implementing integrations.
- **AgentKit/CDP:** the reviewed docs did not consistently expose page update dates. Claims in this report are scoped to the retrieved official pages and [official repository](https://github.com/coinbase/agentkit), not to an inferred package version.
- **Nevermined:** the reviewed pages did not expose per-page update dates. Product and integration claims are scoped to the retrieved [official documentation](https://docs.nevermined.app/docs/getting-started/welcome); production terms, fees, supported rails, and hosted-service behavior should be rechecked before implementation.
- **x402:** the benchmark targets the official [v2 specification](https://github.com/x402-foundation/x402/blob/main/specs/x402-specification-v2.md). Provider network, asset, scheme, and extension support remains provider-specific; the [network/token guide](https://docs.x402.org/core-concepts/network-and-token-support) should be rechecked for each deployment.
- **ERC-8004:** the canonical [EIP](https://eips.ethereum.org/EIPS/eip-8004) remained Draft at retrieval. Draft semantics and deployments can change; Lucid must not freeze undocumented assumptions.
- **A2A:** the benchmark targets the published [1.0 line](https://a2a-protocol.org/latest/announcing-1.0/) and latest formal docs. Patch releases do not replace the need to state Lucid’s implemented major/minor binding and conformance result.
- **AP2:** the benchmark targets [v0.2](https://github.com/google-agentic-commerce/AP2/releases/tag/v0.2.0). Some commerce operations and retention/delegation details remain outside the current formal [specification](https://ap2-protocol.org/ap2/specification/); the report does not infer production readiness.
- **MPP:** the core [Payment Authentication draft](https://datatracker.ietf.org/doc/draft-ryan-httpauth-payment/) is an individual Internet-Draft, not an endorsed IETF standard, and was due to expire 19 September 2026 unless updated. The local package resolved `mppx` 0.1.1 while the official registry exposed [0.8.12](https://registry.npmjs.org/mppx/0.8.12) on the audit date. Operational maturity or compatibility of a particular SDK or payment method must not be generalized into standards maturity.
- **No first-party page located:** no official core x402/ERC-8004 page was located for Mastra, OpenAI Agents SDK, or LangGraph during this review. This is a documentation-evidence statement, not a claim that integrations cannot be built.

## Final assessment

Lucid does not need more top-level claims. It needs a smaller set of claims that are mechanically true, versioned, and demonstrated through the entire paid-service lifecycle.

The defensible market position is strong: agent frameworks decide what work to do; wallets and protocols move value; **Lucid turns that decision and payment into a reliable application transaction**. The documentation will support that position when:

- every protocol name carries an exact conformance boundary;
- every public API example is compiled against its declared release;
- every paid path explains failure, durability, evidence, and recovery;
- every deployment path ends in a verified canary transaction;
- every security claim names the trust boundary and test;
- every hosted-product destination is either a complete public contract or clearly absent; and
- every major agent framework has a tested composition guide showing that Lucid is the commerce layer, not a replacement for the framework.

That work should begin with A2A, wallet/identity reference, AP2/MPP/ERC maturity labels, and documentation execution coverage. Until those are corrected, expanding the site would increase the surface area of uncertainty rather than the clarity of the product.
